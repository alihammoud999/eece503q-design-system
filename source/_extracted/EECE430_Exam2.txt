Detailed Explanations:
1. C — The Agile Manifesto explicitly values individuals and interactions over processes and tools.
2. B — In TDD, a new test should fail initially. If it passes without new code, either the functionality already
exists or the test is not correctly checking the new requirement.
3. B — Agile is suited for small/medium products for sale and custom systems with customer commitment
and few external regulations.
4. C — Lo-Fi UI sketches help customers participate in design. They are less intimidating, easier to change,
and focus on interaction rather than visual polish.
5. B — Stories estimated at 25+ are too complex. The team should divide them into simpler stories that can
be estimated and implemented more reliably.
6. B — The Product Owner identifies features, prioritizes requirements, and continuously reviews the product
backlog.
7. C — Sprints are timeboxed and never extended. Unfinished items go back to the product backlog for future
sprints.
8. B — Sustainable pace means no excessive overtime, as it reduces code quality and productivity in the
medium term.
9. B — The four backlog activities are: refinement (analyze/refine PBIs), estimation (assess effort),
prioritization (reorder), and creation (add new items).
10. B — 5-8 is large enough for skill diversity but small enough for effective informal communication. Teams
are self-organizing and take collective responsibility.
11. C — When user numbers change quickly, the architecture must support scaling. This is the 'number of
users' design issue.
12. B — Availability requires redundant components and fault tolerance. The trade-off is added complexity,
cost, and increased chance of bugs/vulnerabilities.
13. B — MVC allows data to change independently of representation. Changes in one representation are
shown in all Views.

14. B — Transaction processing systems use: I/O processing -> application logic -> transaction manager ->
database.
15. C — Cross-cutting concerns like security affect ALL layers. Distributing them across layers creates
inter-layer dependencies, making post-design changes difficult.
16. B — IaaS provides infrastructure (compute, network, storage). PaaS adds libraries, frameworks, and
services like databases on top of infrastructure.
17. B — Multi-tenant systems use logical isolation (tenant IDs). A single bug or breach can expose all
tenants' data — this is a known disadvantage.
18. C — Update management is a provider benefit: they control updates, all customers get them at the same
time, reducing maintenance of multiple versions.
19. D — Multi-cloud uses multiple cloud providers, avoiding data lock-in and single points of failure. But you
must handle API differences and cross-cloud migration.
20. C — Containers are smaller, start faster, and share the host OS kernel, making them more
resource-efficient than VMs.
21. C — DevOps eliminates separate teams. The three principles are: everyone responsible for everything,
automate everything, measure first change later.
22. B — When the same lines are changed in different branches, Git signals a conflict. Developers must
collaborate to resolve it manually.
23. B — CI builds and tests on the dev machine. CD additionally tests in a production-replica (staged)
environment, catching environment-specific bugs.
24. B — CM tools like Puppet/Chef read infrastructure models and automatically provision servers, install
software, and push changes consistently.
25. B — A/B testing deploys new version on some servers, old on others, and diverts customers to compare
feature usage and effectiveness.
26. B — ACID bundles updates into one atomic unit. This is impractical across microservices since each has
its own database — you'd need distributed transactions.
27. C — The API gateway does NOT eliminate the need for authentication in individual services. It insulates
the client, handles routing, and enables service decomposition changes.
28. B — The common closure principle: elements that change together should be in the same service, so
most new requirements affect only one service.
29. C — The simplest way to report microservice failures is using HTTP status codes, which indicate success
or failure of requests.
30. C — Phishing attacks trick users into clicking fake links and entering credentials on fraudulent sites that
mimic legitimate ones.
31. B — Keys are changed regularly for security. Archived data may use older keys, so KMS must keep
timestamped versions to decrypt backups/archives when needed.
32. B — HTTPS prevents traffic monitoring for cookies, short timeouts limit the window of attack, and MFA
confirms identity for sensitive actions.
33. B — GDPR requires a plain-language summary of the privacy policy on the website, not buried in long
legal documents that nobody reads.
